# Spec: exemplar

The Exemplar is the contract for compendium content authored by a game system. It is
consumed by dtk-promptuarium's build-time compiler (Node.js CLI), which validates
Exemplar YAML files, then compiles them to Foundry LevelDB compendium packs.
Exemplars are rules-layer definitions — they describe what a thing is and what it
grants, not how it appears as a Foundry document (the compiler decides that).

**Revision note**: Three new `kind` values added — `"rule"`, `"sequence"`, `"action"`.
A new grant type `"rule-modifier"` is added. The Grant union and type guards updated
to include these (see design.md Decisions 6, 9, 12, 16).

## ADDED Requirements

### Requirement: Base Exemplar envelope

Every Exemplar, regardless of kind, SHALL carry:

- `id` — stable kebab-case slug, unique within the corpus (e.g., `"elf-species"`)
- `name` — human-readable name (localizable string)
- `kind` — discriminator string selecting the Exemplar subtype (see Kind requirement)
- `summary` — one-sentence plain-text description
- `description` — longer prose (MAY contain markdown); MAY be auto-generated by
  Promptuarium at build time when absent (applies to rule, sequence, action kinds)
- `source` — publication source string (e.g., `"core-rulebook"`)
- `tags` — array of string tags for corpus queries
- `version` — semver string for the Exemplar definition

Optional envelope fields: `img` (image path string), `prerequisites` (array of Condition
predicates using the same Condition type as Forma), `flags` (Record<string, unknown>
escape hatch), `grants` (array of Grant nodes — see Grant model requirement).

#### Scenario: Minimal valid Exemplar

- **WHEN** an Exemplar has id, name, kind, summary, description, source, tags, version
- **THEN** the Zod validator accepts it

#### Scenario: Missing id rejected

- **WHEN** an Exemplar omits `id`
- **THEN** the Zod validator returns an error

#### Scenario: Invalid semver version rejected

- **WHEN** `version: "not-a-version"` is declared
- **THEN** the Zod validator returns an error

---

### Requirement: Kind discriminator

An Exemplar's `kind` field SHALL be one of the following values, each selecting a
distinct schema extension:

- `"species"` — a playable species / ancestry
- `"archetype"` — top-tier class layer
- `"discipline"` — mid-tier class layer (requires `parent` archetype reference)
- `"vocation"` — bottom-tier class layer (requires `parent` discipline reference)
- `"item"` — a piece of gear, equipment, or inventory item
- `"background"` — character background / origin story
- `"origin"` — mechanical origin / starting package
- `"feature"` — a standalone feature, talent, or ability
- `"rule"` — an atomic action procedure (pool, mechanic, consequences)
- `"sequence"` — an ordered series of rule or sequence steps
- `"action"` — a UI-layer entry point over a sequence (targeting, cost, icon)

#### Scenario: Valid kind accepted

- **WHEN** `kind: "species"` is declared
- **THEN** the validator routes to the species subschema

#### Scenario: Rule kind accepted

- **WHEN** `kind: "rule"` is declared with required rule fields
- **THEN** the validator routes to the rule subschema

#### Scenario: Invalid kind rejected

- **WHEN** `kind: "monster"` is declared
- **THEN** the Zod validator returns an error listing valid kind values

#### Scenario: Discipline requires parent

- **WHEN** `kind: "discipline"` is declared without a `parent` field
- **THEN** the Zod validator returns an error

### Requirement: Grant model — modifier node

A `modifier` grant SHALL carry:

- `type: "modifier"` — discriminator
- `target` — a slug string referencing an attribute, skill, or derived stat
- `value` — a number OR a formula string (e.g., `"@tier + 1"`)
- `mode` — `"add"` | `"multiply"` | `"override"` (default: `"add"`)

A modifier compiles to a Foundry Active Effect.

#### Scenario: Numeric modifier accepted

- **WHEN** `{ type: "modifier", target: "body", value: 2, mode: "add" }` is declared
- **THEN** the validator accepts it

#### Scenario: Formula string modifier accepted

- **WHEN** `{ type: "modifier", target: "defense", value: "@tier + 1" }` is declared
- **THEN** the validator accepts it

#### Scenario: Invalid mode rejected

- **WHEN** `mode: "subtract"` is declared
- **THEN** the Zod validator returns an error

---

### Requirement: Grant model — reference node

A `reference` grant SHALL carry:

- `type: "reference"` — discriminator
- `ref` — an Exemplar id slug string referencing another Exemplar in the corpus
- `quantity` — positive integer (default: 1)
- `equipped` — boolean (default: false) — whether the granted item starts equipped
- `overrides` — optional Record<string, unknown> for per-grant field overrides

#### Scenario: Reference grant accepted

- **WHEN** `{ type: "reference", ref: "stealth-skill" }` is declared
- **THEN** the validator accepts it

#### Scenario: Zero quantity rejected

- **WHEN** `quantity: 0` is declared
- **THEN** the Zod validator returns an error

---

### Requirement: Grant model — choice node

A `choice` grant SHALL carry:

- `type: "choice"` — discriminator
- `label` — player-facing prompt string
- `count` — how many selections the player makes (positive integer, default: 1)
- `unique` — boolean, whether the same option can be selected multiple times (default: true)
- `resolveAt` — `"author"` | `"build"` | `"character"` (default: `"character"`)
- `pool` — one of:
  - `{ list: string[] }` — explicit list of Exemplar id slugs
  - `{ query: { kind?: string, tags?: string[], where?: string } }` — corpus query

Each option in a `list` pool MAY carry further nested grants, making `choice`
the recursive node in the Grant model.

#### Scenario: List choice accepted

- **WHEN** `{ type: "choice", label: "Pick a skill", pool: { list: ["stealth", "perception"] } }` is declared
- **THEN** the validator accepts it

#### Scenario: Query choice accepted

- **WHEN** `{ type: "choice", label: "Pick a species feature", pool: { query: { kind: "feature", tags: ["elf"] } } }` is declared
- **THEN** the validator accepts it

#### Scenario: Invalid resolveAt rejected

- **WHEN** `resolveAt: "runtime"` is declared
- **THEN** the Zod validator returns an error

#### Scenario: Zero count rejected

- **WHEN** `count: 0` is declared
- **THEN** the Zod validator returns an error

---

### Requirement: Grant model — rule-modifier node

A `rule-modifier` grant overrides specific fields on a referenced Rule or Sequence
Exemplar. It is used by class features, species traits, and other Exemplars that need
to patch a base rule (e.g., a dwarf's recoil compensation reducing the recoil penalty
in a ranged attack rule).

A `rule-modifier` grant SHALL carry:

- `type: "rule-modifier"` — discriminator
- `ref` — the `id` of the target Rule or Sequence Exemplar
- `override` — a Record<string, unknown> mapping dot-notation field paths to
  replacement values. Paths reference fields within the target rule's schema
  (e.g., `"threshold"`, `"on_tier.hit.damage"`, `"pool"`).

The override is applied at build time by Promptuarium when compiling the actor's
effective rule set. Cross-reference validity (does `ref` exist and is it a rule or
sequence?) is the compiler's responsibility, not the schema's.

#### Scenario: Threshold override accepted

- **WHEN** `{ type: "rule-modifier", ref: "ranged-attack", override: { "threshold": 4 } }` is declared
- **THEN** the validator accepts it

#### Scenario: Pool expression override accepted

- **WHEN** `{ type: "rule-modifier", ref: "ranged-attack", override: { "pool": "@agility + @skill.firearms + 2" } }` is declared
- **THEN** the validator accepts it

#### Scenario: Missing ref rejected

- **WHEN** `{ type: "rule-modifier", override: { "threshold": 4 } }` is declared without `ref`
- **THEN** the Zod validator returns an error

#### Scenario: Empty override rejected

- **WHEN** `{ type: "rule-modifier", ref: "ranged-attack", override: {} }` is declared
- **THEN** the Zod validator returns an error — override must have at least one entry

---

### Requirement: Grant union

The `grants` array on an Exemplar (and within choice options) SHALL accept any mix of
`modifier`, `reference`, `choice`, and `rule-modifier` nodes, discriminated by the
`type` field. The Grant type SHALL be exported as a discriminated union:

```
Grant = Modifier | Reference | Choice | RuleModifier
```

#### Scenario: Mixed grants accepted

- **WHEN** an Exemplar's grants array contains a modifier, a reference, a choice, and a rule-modifier
- **THEN** the validator accepts all four entries

#### Scenario: Unknown grant type rejected

- **WHEN** a grant declares `type: "bonus"`
- **THEN** the Zod validator returns an error

---

### Requirement: Class layer parent references

Exemplars with `kind: "discipline"` SHALL declare `parent` as a string (the `id`
of a `kind: "archetype"` Exemplar). Exemplars with `kind: "vocation"` SHALL declare
`parent` as a string (the `id` of a `kind: "discipline"` Exemplar). Cross-reference
validity (does the parent actually exist in the corpus?) is the compiler's
responsibility, not the schema's.

#### Scenario: Discipline with parent accepted

- **WHEN** `{ kind: "discipline", parent: "street-samurai" }` is declared
- **THEN** the validator accepts it

#### Scenario: Vocation without parent rejected

- **WHEN** `kind: "vocation"` is declared without a `parent` field
- **THEN** the Zod validator returns an error

### Requirement: kind: rule

An Exemplar with `kind: "rule"` describes an atomic action procedure — what dice to
roll and what the consequences of each outcome tier are. It replaces the TypeScript
`poolBuilder` and `onComplete` hooks that were in the prototype's pipeline.

A rule Exemplar SHALL carry:

- `ritus` — string: the Ritus id this rule targets (e.g., `"sr5e"`)
- `pool` — string: expression for the dice pool (e.g., `"@agility + @skill.firearms - @recoil"`)
- `on_tier` — Record<string, TierConsequence>: keyed by tier name (must match Ritus tiers
  or the per-rule tier overrides). Each TierConsequence may declare:
  - `damage?` — expression string (e.g., `"@net_hits + @weapon.damage"`)
  - `chain?` — id of a named chain defined in `chains`, OR a Rule Exemplar id
  - `effect?` — Exemplar id of an effect to apply
  - `message?` — localizable string for chat output

Optional rule fields:
- `opposed` — string: opposition pool expression (for opposed rolls)
- `mechanic` — mechanic enum value: overrides the Ritus system-wide default for this rule
- `threshold` — positive integer: overrides the Ritus system-wide threshold for this rule
- `tiers` — partial tiers object: overrides Ritus tier breakpoints for this rule
- `chains` — Record<string, ChainDef>: named chained roll definitions. Each ChainDef carries:
  - `pool` — expression string
  - `mechanic?` — optional override
  - `resolves` — `"actor"` | `"target"` — who rolls

The `description` field in the base envelope MAY be absent; Promptuarium will generate
it from the DSL at build time using Codex display names.

#### Scenario: Minimal rule accepted

- **WHEN** a rule Exemplar declares `ritus`, `pool`, and `on_tier` with at least one tier entry
- **THEN** the Zod validator accepts it

#### Scenario: Rule with threshold override accepted

- **WHEN** a rule declares `threshold: 4` alongside Ritus id `"sr5e"` (which has threshold 5)
- **THEN** the validator accepts it as a valid per-rule threshold override

#### Scenario: Rule with opposed pool accepted

- **WHEN** `opposed: "@target.reaction + @target.intuition"` is declared
- **THEN** the validator accepts it

#### Scenario: Rule without pool rejected

- **WHEN** a rule Exemplar omits the `pool` field
- **THEN** the Zod validator returns an error

#### Scenario: Rule without ritus rejected

- **WHEN** a rule Exemplar omits the `ritus` field
- **THEN** the Zod validator returns an error

#### Scenario: Rule with named chain accepted

- **WHEN** `chains: { "soak": { pool: "@body", mechanic: "pool-count", resolves: "target" } }` is declared
- **THEN** the validator accepts it

### Requirement: kind: sequence

An Exemplar with `kind: "sequence"` describes an ordered series of steps, each
referencing a Rule or another Sequence. Sequences are the composition primitive —
a melee attack might be a sequence of ranged-attack-rule → soak-sequence →
apply-damage-rule.

A sequence Exemplar SHALL carry:

- `steps` — array of SequenceStep (min 1). Each SequenceStep SHALL carry:
  - `id` — string: stable identifier for this step (used in cross-step references)
  - `rule` OR `sequence` — exactly one of: a Rule Exemplar id or a Sequence Exemplar id
  - `actor` — `"initiator"` | `"target"` | `"gm"` | `"all"`: who performs this step
  - `condition?` — optional Condition predicate (same type as Forma Condition); step is
    skipped if the condition is false; skipped steps write a null sentinel to `@steps.{id}`
  - `await?` — optional await spec (see below): suspend execution at this step for
    player input before continuing
  - `inputs?` — optional Record<string, string>: expression map injecting values into
    this step's context (e.g., `{ "pool_bonus": "@steps.cover-roll.net_hits" }`)

A `await` spec SHALL carry:
- `type: "player-decision"` — the only supported await type
- `choices` — array of Choice objects, each with `id: string` and `label: string`
- `timeout?` — positive integer seconds (default: none; GM must resolve)
- `default?` — choice id selected if timeout expires with no player action

Cross-step references use the syntax `@steps.{id}.{field}` (e.g.,
`@steps.initial-attack.net_hits`). A reference to a skipped step resolves to null.

`kind: "sequence"` is self-referential — a step may reference another sequence by id,
making this a recursive type. The Zod schema for SequenceStep SHALL use `z.lazy()` at
the sequence reference to avoid infinite expansion.

#### Scenario: Minimal sequence accepted

- **WHEN** a sequence Exemplar declares `steps` with one step referencing a rule id, actor `"initiator"`
- **THEN** the Zod validator accepts it

#### Scenario: Multi-step sequence with cross-reference accepted

- **WHEN** step 2 declares `inputs: { "soak_threshold": "@steps.attack-roll.net_hits" }` referencing step 1
- **THEN** the Zod validator accepts the sequence

#### Scenario: Step with await accepted

- **WHEN** a step declares `await: { type: "player-decision", choices: [{ id: "dodge", label: "Dodge" }, { id: "block", label: "Block" }] }`
- **THEN** the validator accepts it

#### Scenario: Await with timeout and default accepted

- **WHEN** `await: { type: "player-decision", choices: [...], timeout: 30, default: "dodge" }` is declared
- **THEN** the validator accepts it

#### Scenario: Step without rule or sequence rejected

- **WHEN** a step declares `actor: "initiator"` but neither `rule` nor `sequence`
- **THEN** the Zod validator returns an error

#### Scenario: Step with both rule and sequence rejected

- **WHEN** a step declares both `rule: "ranged-attack"` and `sequence: "full-attack-sequence"`
- **THEN** the Zod validator returns an error — exactly one is required

#### Scenario: Empty steps array rejected

- **WHEN** `steps: []` is declared
- **THEN** the Zod validator returns an error

### Requirement: kind: action

An Exemplar with `kind: "action"` is a thin UI-layer entry point over a sequence. It
carries targeting configuration, cost declaration, and display metadata. Actions appear
in the action menu rendered by dtk-systema; clicking one triggers dtk-systema's target
selection and context building before handing off to dtk-alea for execution.

An action Exemplar SHALL carry:

- `sequence` — string: the id of a `kind: "sequence"` Exemplar to execute
- `targeting` — TargetingSpec (see below)

Optional action fields:
- `cost` — ActionCost: declares resource costs consumed on use
- `hint` — string: short UI label override (defaults to `name`)
- `icon` — string: path to an icon image
- `condition` — Condition predicate: if false, the action is greyed out in the menu
  but still visible (not hidden)

**TargetingSpec** SHALL carry:

- `mode` — `"token"` | `"self"` | `"area"` | `"none"`:
  - `"token"`: player selects one or more tokens on the canvas
  - `"self"`: the initiating actor is automatically the target (no selection UI)
  - `"area"`: player places a template on the canvas; all tokens in the area are targets
  - `"none"`: no target; useful for self-affecting or environmental actions

For `mode: "token"` only:
- `min?` — positive integer: minimum tokens required (default: 1)
- `max?` — positive integer: maximum tokens selectable (default: 1); set higher for multi-target
- `filter?` — expression string: token filter predicate (e.g., `"@token.disposition == 'hostile'"`)
- `execution?` — `"per-target"` | `"once"`:
  - `"per-target"` (default): the sequence executes once per selected token
  - `"once"`: the sequence executes once with all selected tokens passed as a list

For `mode: "area"` only:
- `shape` — `"circle"` | `"cone"` | `"ray"` | `"rect"`: template shape
- `size` — positive number: radius (circle), length (cone/ray/rect) in grid units
- `width?` — positive number: width for ray and rect shapes

**ActionCost** MAY declare any of:
- `actionPoints?` — positive integer
- `freeActions?` — positive integer
- `ammo?` — positive integer
- `custom?` — Record<string, number> for system-specific resource keys

#### Scenario: Self-targeting action accepted

- **WHEN** an action declares `sequence: "heal-self-seq"` and `targeting: { mode: "self" }`
- **THEN** the Zod validator accepts it

#### Scenario: Token targeting with min/max accepted

- **WHEN** `targeting: { mode: "token", min: 1, max: 3, execution: "per-target" }` is declared
- **THEN** the validator accepts it

#### Scenario: Area targeting with shape accepted

- **WHEN** `targeting: { mode: "area", shape: "cone", size: 6, width: 3 }` is declared
- **THEN** the validator accepts it

#### Scenario: Area targeting without shape rejected

- **WHEN** `targeting: { mode: "area", size: 6 }` is declared without `shape`
- **THEN** the Zod validator returns an error

#### Scenario: Token min exceeding max rejected

- **WHEN** `targeting: { mode: "token", min: 3, max: 1 }` is declared
- **THEN** the Zod validator returns an error

#### Scenario: Action without sequence rejected

- **WHEN** an action Exemplar omits the `sequence` field
- **THEN** the Zod validator returns an error

#### Scenario: Action with condition renders greyed

- **WHEN** an action declares `condition: { field: "@actor.ammo", op: "gt", value: 0 }`
- **THEN** the validator accepts it; dtk-systema is responsible for greying it when false

#### Scenario: Action with cost accepted

- **WHEN** `cost: { actionPoints: 1, ammo: 6 }` is declared
- **THEN** the validator accepts it

### Requirement: Exemplar Zod validator

`@dtk/types/exemplar` SHALL export an `ExemplarSchema` Zod discriminated union schema
and a derived `Exemplar` TypeScript type. Sub-schemas for each kind SHALL be exported
individually (`SpeciesSchema`, `ArchetypeSchema`, `RuleSchema`, `SequenceSchema`,
`ActionSchema`, etc.) for use by the compiler's per-kind validation passes.

Recursive types SHALL use `z.lazy()`:
- The `choice` grant's option grants (existing)
- The `sequence` step's `sequence` reference (new — self-referential)

#### Scenario: Valid species Exemplar parses cleanly

- **WHEN** `ExemplarSchema.parse(validSpecies)` is called
- **THEN** it returns the object typed as `Exemplar` (narrowed to species)

#### Scenario: Valid rule Exemplar parses cleanly

- **WHEN** `ExemplarSchema.parse(validRule)` is called with required rule fields
- **THEN** it returns the object typed as `Exemplar` (narrowed to rule)

#### Scenario: Valid sequence Exemplar parses cleanly

- **WHEN** `ExemplarSchema.parse(validSequence)` is called with a two-step sequence
- **THEN** it returns the object typed as `Exemplar` (narrowed to sequence)

#### Scenario: Deeply nested choice resolves without stack overflow

- **WHEN** an Exemplar with a three-level nested choice tree is parsed
- **THEN** the validator completes without error

#### Scenario: Recursive sequence resolves without stack overflow

- **WHEN** a sequence with a step that references another sequence is parsed
- **THEN** the validator completes without error

### Requirement: Exemplar type guard

`@dtk/types/exemplar` SHALL export an `isExemplar(value: unknown): value is Exemplar`
type guard and per-kind guards for all eleven kinds:

`isSpecies`, `isArchetype`, `isDiscipline`, `isVocation`, `isItem`, `isBackground`,
`isOrigin`, `isFeature`, `isRule`, `isSequence`, `isAction`

All guards are backed by the corresponding sub-schema's `safeParse`. No guard throws
on invalid input.

#### Scenario: Species guard narrows to species type

- **WHEN** `isSpecies(validSpecies)` is called
- **THEN** it returns `true` and TypeScript narrows to the species subtype

#### Scenario: Rule guard narrows to rule type

- **WHEN** `isRule(validRule)` is called
- **THEN** it returns `true` and TypeScript narrows to the rule subtype

#### Scenario: Cross-kind guard returns false

- **WHEN** `isArchetype(validSpecies)` is called
- **THEN** it returns `false`

#### Scenario: isAction returns false for sequence

- **WHEN** `isAction(validSequence)` is called
- **THEN** it returns `false`
